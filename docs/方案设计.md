## **1. 架构总览与核心分流总线**

当用户点击系统分享并激活 Share Extension 时，插件的 ShareViewController 在 viewDidLoad 的前 50 毫秒内，通过**宿主特征嗅探**与**文件物理尾部二进制扫描**，将逻辑划分为 **2大物理前置特征** 与 **4大精简业务通路**：

 

<br />

```
                               收到传入的文件 tempFile
                                         |
                       +-----------------+-----------------+
                       |                                   |
            [ 检测到：尾部存在元数据 ]           [ 检测到：尾部无元数据 ]
                       |                                   |
           +-----------+-----------+           +-----------+-----------+
           |                       |           |                       |
     ( 宿主为外部/保险箱 )    ( 宿主为 WPS )   ( 宿主为外部/微信 )   ( 宿主为 WPS )
           |                       |           |                       |
     【通路 1：统一看密】     【通路 2：纯看未改】 【通路 3：野生文件】  【通路 4：正常写回】

```

<br />

 

## **2. 4 大精简业务通路完整路径规范**

### **🛠️ 通路 1：统一查看密码线（核心记账防线 · 原组合 A & E 合并）**

● **适用场景**：用户在微信/QQ 中收到带有元数据的文件（读场景① & ②），或者在系统“文件 App”的保险箱目录中直接对有元数据的文件点击分享（读场景③）。

● **用户心智**：**“我只是想用插件临时查个密码，然后我自己去手动打开它。”**

● **核心动作（物理轻量、数据有心）**：

○ **元数据提取**：扫描 tempFile 尾部，成功提取出有效的企业资产 UID。

○ **界面交互**：插件原地进入 **【统一查密画布】**（高亮、特大字号直接在屏幕中央砸出该 UID 对应的解密密码，并展示标准资产卡片）。界面只提供一个动作大按钮：**\[ 复制密码并退场 ]**。

○ **无感记账（至关重要）**：在用户点击“复制”的瞬间，插件在后台花费 1 毫秒通过 **REPLACE INTO** 语法将该文件的 \[UID : 密码句柄 : 当前时间戳] 写入 App Group 共享数据库，并显式标记 **is\_local\_vault = 0**（表明物理文件此时不在保险箱）。**这一步是为通路 4 的全自动撞击埋下绝对完美的垫脚石**。

○ **物理零动作**：**绝对不调用 copyItem 复制或搬运文件**。文件依然各自安全地躺在它们原来的地方（微信的在微信，保险箱的在保险箱）。

○ **智能退场与回弹**：内存级将密码挂载到系统剪贴板（60秒销毁）后，插件瞬间主动调用 completeRequest 自动闪退。用户瞬间视觉无缝回弹到原宿主应用（微信或文件 App），直接双击文件拉起 WPS 粘贴即可。

### **🛠️ 通路 2：WPS 纯看未改写回线（组合 8）**

● **适用场景**：用户在 WPS 内打开了一个带有元数据的文件，但**纯阅读、未做任何内容编辑修改**，随后直接在 WPS 内点击了分享发回插件。

● **物理特征**：由于 WPS 没有触发真正的重写保存，文件的 ZIP 结构未被破坏，**尾部元数据奇迹般地保留了下来**。

● **核心动作**：

○ **闪电免撞击**：判定宿主来自于 WPS，且扫描发现尾部存在有效元数据。立即**免去耗时的后台密码撞击**，直接提取尾部 UID。

○ **账本反查**：用该 UID 去 SQLite 数据库反查出对应的老资产记录。

○ **精准直通**：界面跳过所有密码拦截，直接原地全屏切换为 **【UI 状态 B：精准同步确认画布】**。提示用户确认覆盖或另存。

### **🛠️ 通路 3：野生资产首次实名登记线（组合 1 & 3）**

● **适用场景**：用户从外部微信、QQ 或 iCloud 目录调入了一个纯粹野生、游离在企业生态之外的、没有任何元数据标记的加密文件。

● **用户心智**：**“把外部野生的加密资产正式收纳并登记落户到企业保险箱中。”**

● **核心动作**：

○ **乱码纠偏**：如果文件名由于微信缓存机制变为乱码（如 \~6fa8b2a3c9），触发 **【真名确认与纠偏画布】（UI 状态 A）**。流式读取前 4 字节魔数 50 4B 03 04 确保为合规 OOXML 格式，浅扫描 ZIP 目录自动关联补齐后缀。

○ **防黑户拦截（黄色画布）**：基于“野生文件不瞎认”原则，立即弹出 **【黄色画布（登记新文件）】**，强制要求用户手动输入该文档在 WPS 内设置的正确密码。

○ **解密合法性验证**：调用 OfficeCryptoVerifier 在后台异步线程校验用户输入的密码能否真正解开 tempFile。

■ *校验失败*：原地平滑降级切换至错误画布，阻断流程。

■ *校验通过*：确认为合法落户资产。在内存中生成全局唯一的 UID\_全新，连同有效密码**追加写入文件尾部**。

○ **首次入库落盘**：将新生成的 UID 映射关系写入数据库（标记 **is\_local\_vault = 1**）。最后采用 NSFileCoordinator 协调的同步 moveItem 操作，将文件安全转正落盘到共享沙盒的 **“安全保险箱”目录** 中，并设置备份排除标记。

### **🛠️ 通路 4：WPS 正常编辑写回线（核心回写流 · 组合 7）**

● **适用场景**：用户在手机 WPS 里面大刀阔斧地修改了原本属于生态内的合规文件，点击保存并分享回插件。

● **物理特征**：由于 WPS 重新编译并打包了 ZIP 压缩流，**文件尾部原本的元数据已经被 WPS 彻底抹空（无元数据状态）**。

● **核心动作**：

○ **后台全自动降维打击（解密撞击）**：**严禁直接弹窗逼用户选**。插件在后台线程（DispatchQueue.global）中，从 SQLite 账本中按时间戳倒序取出 **前 5 条活跃记录的密码队列**。

○ **撞击结果二元分流处理**：

■ **情况 A（撞击成功 ·“正规军洗澡回来了”）**：

● 由于**通路 1** 已经在用户查密时悄悄执行了“无感记账”，此时密码队列中**必然 100% 包含并命中**该文件的解密密码！

● 代码通过撞中的密码，瞬间反查出对应的 UID\_老 和原始 file\_name。

● **零交互直通**：直接跳过强制密码输入，原地全屏切换为 **【UI 状态 B：精准同步确认画布】**。

● 用户点击 **\[确定同步覆盖]** ➔ 调用 NSFileCoordinator 内核锁重试逻辑，删除保险箱老文件，将回传文件更名覆盖转正，并将数据库中的 is\_local\_vault 状态从 0 修正为 1。**保持原 UID 基因在云端和本地绝对不变**。

■ **情况 B（所有密码均撞击失败 · 用户在 WPS 里顺手改了密码，或完全手创新资产）**：

● 此时技术手段彻底失效，正式降级切入 **【黄色画布（密码强制捕获与外部资产归宿确认）】**。

● 用户在画布中输入最新密码，验证通过后，全屏提供两个大按钮供用户肉眼决策：

○ **\[ 登记为本地新文件 ]**：生成全新的 UID\_全新 追加写入尾部并落盘。

○ **\[ 关联并覆盖本地已有文件 ]**：平滑切换至 **【状态 D：基于 SwiftUI 惰性高性能滑动认领画布】**，由用户人工指定要覆盖替换哪一个老资产，指定后注入老 UID 并跨卷覆盖。

## **3. 数据库结构与多进程全自治维护规范**

### **3.1 核心轻量账本表结构 (file\_mapping\_table)**

SQL

<br />

```
CREATE TABLE IF NOT EXISTS file_mapping_table (
    uid TEXT PRIMARY KEY,                 -- 文件的唯一加密基因 UID (主键)
    file_name TEXT NOT NULL,              -- 标准化小写文件名 (用于辅助模糊匹配)
    password_hash TEXT NOT NULL,          -- 经托管的密码密文句柄/解密盐值
    last_access_time INTEGER NOT NULL,    -- 最后一次读取/写入的时间戳 (用于 LRU 淘汰和Top 5排序)
    file_size INTEGER NOT NULL,           -- 物理文件大小 (用于辅助特征判定)
    is_local_vault INTEGER DEFAULT 0      -- 核心标记：0 表示文件在外部(如微信)纯记账；1 表示文件已落盘在保险箱
);
CREATE INDEX IF NOT EXISTS idx_filename_size ON file_mapping_table(file_name, file_size);
CREATE INDEX IF NOT EXISTS idx_access_time ON file_mapping_table(last_access_time);

```

<br />

 

### **3.2 刚性双向保洁总线（谁来删，怎么删）**

由于 Share Extension 插件进程用完即被系统销毁，**重度事务的大扫除维护全权交由主 App 在后台默默执行**：

1.  **僵尸索引自动清洗（主 App 异步执行）**： 主 App 每次启动或由后台唤醒时，利用 FileManager 异步扫描一遍公共沙盒（安全保险箱）目录。如果发现数据库中某条记录 **is\_local\_vault = 1**，但在公共目录下已经找不到它的物理实体了（说明用户直接在系统的“文件 App”里把文件长按删除了）。主 App 立即执行批量删除，剔除僵尸索引：

SQL

<br />

```
DELETE FROM file_mapping_table WHERE uid = ?;

```

<br />

 

2.  **磁盘空间 LRU 硬性清理（主 App 限额控制）**： 当共享保险箱目录总占用超过硬性阈值（如 1GB）时，主 App 计算超额空间，从数据库中按照 last\_access\_time 升序（最久没人访问的文件）查询出记录：

SQL

<br />

```
SELECT uid, file_name FROM file_mapping_table WHERE is_local_vault = 1 ORDER BY last_access_time ASC;

```

<br />

 

主 App 循环删除这批最冷门文件的本地物理实体，并同步执行数据库 DELETE，直到磁盘空间回落到安全线内。

### **3.3 多进程 WAL 并发硬化控制**

为了绝对防止主 App 和 插件在同一毫秒读写数据库冲突抛出 SQLITE\_BUSY 错误，底层连接强制注入以下两条刚性约束：

● **开启 WAL 模式**：初始化 SQLite 链接时，强制执行 PRAGMA journal\_mode=WAL;。

● **配置 2 秒 Busy Timeout 避让重试锁**：

Swift

<br />

```
// Swift 底层连接硬化：如果遇到多进程锁冲突，最长自动等待重试 2000 毫秒，2秒内有空隙就无感插队写入
sqlite3_busy_timeout(dbConnection, 2000)

```

<br />

 

 

## **4. Office 密码验证技术实现**

### **4.1 支持的加密格式**

当前实现支持 **OLE2 复合文档格式** 的 **Standard Encryption (version 4.2)**，这是 Office 2007+ 使用的加密标准。

### **4.2 OLE2 复合文档解析**

实现了完整的 OLE2 复合文档解析器（OLE2Parser），包含以下核心功能：

- **文件头解析**: 读取扇区大小、迷你扇区大小、FAT 扇区数、根目录扇区等关键参数
- **DIFAT 读取**: 支持扩展 DIFAT 链读取，处理大文件场景
- **FAT 读取**: 从 DIFAT 条目读取完整的 FAT 链
- **目录项遍历**: 解析目录项结构（名称、类型、起始扇区、大小），支持存储/流/根目录类型
- **流读取**: 支持普通流和迷你流读取

### **4.3 EncryptionInfo 解析**

从 OLE2 文档中提取 `EncryptionInfo` 流，解析以下关键字段：

| 字段 | 说明 |
|------|------|
| versionMajor/minor | 加密版本（4.2 表示 Standard Encryption） |
| headerSize | Header 结构大小 |
| algId | 加密算法 ID（0x660E = AES-128） |
| algIdHash | 哈希算法 ID（0x8004 = SHA-1） |
| keySize | 密钥大小（128 位） |
| salt | 盐值（16 字节） |
| encryptedVerifier | 加密的验证值（16 字节） |
| encryptedVerifierHash | 加密的验证哈希值（32 字节） |

### **4.4 密钥派生算法（ECMA-376 标准）**

严格按照 ECMA-376 标准实现密钥派生：

```
1. 初始哈希: H0 = SHA-1(salt + password_UTF16LE)

2. 迭代哈希: H(i) = SHA-1(iterator(i) + H(i-1))，共迭代 50000 次
   - iterator(i): 4 字节小端序表示的迭代次数

3. 最终哈希: finalHash = SHA-1(H50000 + blockKey)，blockKey = [0,0,0,0]

4. fillAndXor 操作:
   - x1 = SHA-1(0x36填充64字节 XOR finalHash)
   - x2 = SHA-1(0x5c填充64字节 XOR finalHash)

5. 最终密钥: AES_KEY = (x1 + x2).prefix(keySize)
```

### **4.5 密码验证流程**

```
输入密码 → 密钥派生 → AES-ECB解密
                              │
              ┌───────────────┴───────────────┐
              ▼                               ▼
    encryptedVerifier                encryptedVerifierHash
              │                               │
              ▼                               ▼
         verifier                      decryptedVerifierHash
              │                               │
              ▼                               │
    SHA-1(verifier)                           │
              │                               │
              ▼                               ▼
      computedHash                   decryptedVerifierHash[0..19]
              │                               │
              └───────────────┬───────────────┘
                              ▼
                        比对结果 → true/false
```
